## 신규 버그 목록

### BUG-E2E-001: WS chief_response 메시지 타이밍 경쟁 조건
- **심각도:** Medium
- **재현:** chief/chat POST 후 즉시 WS 리스닝 시작 시, LLM 응답이 리스너 등록 전에 도달할 수 있음
- **재현 빈도:** 10/10 (모든 라운드의 첫 chat 요청에서 관찰)
- **원인:** HTTP 응답(`{status:"processing", messageId}`) 수신 후 WS 리스너를 등록하는데, LLM이 매우 빠르면 WS 메시지가 먼저 도착
- **영향도:** 클라이언트가 응답을 놓칠 수 있음. 현재 웹 프론트엔드는 WS를 먼저 연결하므로 실제 사용에서는 영향 적음
- **수정상태:** ⚠️ 미수정 (웹 클라이언트에서는 문제없으나, API 클라이언트 사용 시 주의 필요)
- **핫픽스 권장순위:** 3

### BUG-E2E-002: Deliverable validation 엔드포인트 null 반환
- **심각도:** Medium
- **재현:** `GET /api/deliverables/:id/validate` 호출 시 대부분 null 또는 빈 응답
- **재현 빈도:** 10/10
- **원인:** `validateWebDeliverable()`이 실제 브라우저 렌더링 검증을 수행하지 않고, deliverable type이 web이 아닌 경우 validation 스킵
- **영향도:** 자동 QA 검증 파이프라인에서 결과 신뢰도 하락
- **수정상태:** ⚠️ 미수정
- **핫픽스 권장순위:** 2

### BUG-E2E-003: 태스크 중복 생성
- **심각도:** Low
- **재현:** Round 7(타이핑 게임), Round 9(로드맵 기획)에서 거의 동일한 제목의 태스크가 2건씩 생성됨
- **재현 빈도:** 2/10 (20%)
- **원인:** Chief LLM이 유사한 액션을 2개 제안하거나, 동일 요청이 중복 전송될 때 중복 필터링 없음
- **영향도:** 리소스 낭비, 에이전트 이중 작업
- **수정상태:** ⚠️ 미수정
- **핫픽스 권장순위:** 4

### BUG-E2E-004: 체인 태스크 완료 추적 불완전
- **심각도:** High
- **재현:** Round 10에서 5단계 체인 중 Chain-2~4가 QC 완료 시점에도 in-progress 상태
- **재현 빈도:** 1/10 (체인 워크플로우 사용 시)
- **원인:** 체인 후속 단계가 이전 단계 완료 이벤트를 수신하는 시점이 느리거나, 에이전트 세션 종료 후 상태 업데이트 지연
- **영향도:** 체인 워크플로우의 전체 완료 시간이 예상보다 길어짐
- **수정상태:** ⚠️ 미수정
- **핫픽스 권장순위:** 1

### BUG-E2E-005: 수정 요청 시 새 태스크 생성 vs 기존 태스크 재실행 일관성 없음
- **심각도:** Low
- **재현:** 일부 라운드(R1)에서는 기존 태스크에 assign_task, 다른 라운드(R8)에서는 새 태스크 생성
- **재현 빈도:** ~50%
- **원인:** Chief LLM의 판단에 따라 다름 — 명시적 정책 없음
- **영향도:** 사용자 예측 가능성 저하, 태스크 이력 추적 복잡
- **수정상태:** ⚠️ 미수정 (설계 결정 필요)
- **핫픽스 권장순위:** 5

### BUG-E2E-006: approval API 응답에 실행 결과 포맷 불명확
- **심각도:** Low
- **재현:** `POST /api/chief/proposal/approve` 응답에 `ok: true, executedActions: [...]` 반환하지만, 클라이언트가 실행 성공 여부를 판별하려면 개별 action의 `result.ok`을 확인해야 함
- **재현 빈도:** 10/10
- **원인:** API 설계 — 최상위에 `success` 필드 없음
- **영향도:** 외부 통합 시 혼란
- **수정상태:** ⚠️ 미수정
- **핫픽스 권장순위:** 6
